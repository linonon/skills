# Self-Evolution v0.4 升級計劃

> 基於 Claude Code 架構分析 + 現有 v0.3.0 程式碼審查
> 日期：2026-03-12

---

## 現狀問題

### v0.3.0 做了什麼
- `analyze.js` — 掃描 JSONL transcript 找 6 種錯誤（tool_error, exec_failure, timeout 等）
- `fix.js` — 分 low/high risk，生成修復 prompt
- `lib/classifier.js` — 風險分級規則
- `lib/validator.js` — config 驗證 + 備份 + 回滾
- `loop` command — 閉環修復骨架

### 根本問題

| 問題 | 說明 |
|------|------|
| **架構矛盾** | Node.js CLI 無法呼叫 `sessions_list`、`sessions_history`、`sessions_spawn`（這些是 agent tool，不是 API） |
| **資料來源斷裂** | 硬編碼讀 `~/.openclaw/agents/main/sessions/*.jsonl`，路徑不一定存在 |
| **Markdown 反模式** | 結構化資料 → 生成 markdown → regex 解析回來，脆弱且多餘 |
| **修復空殼** | `fix.js` 生成 prompt 但無法實際執行 `sessions_spawn`，loop command 空轉 |
| **無去重** | 同個錯誤 retry 多次會被重複計算 |
| **無時間窗口** | 沒有「最近 24h」或「上次分析之後」的概念 |

## 架構決策：SKILL.md 驅動（放棄 Node.js CLI）

**核心改變：** 從 Node.js 腳本改為 SKILL.md 指令，讓 agent 用原生 tools 執行。

**理由：**
1. Agent 天生能呼叫 `sessions_list`、`sessions_history`、`sessions_spawn`
2. 現有 Node.js 程式碼大部分在做 agent 本來就會做的事（掃描 JSON、分類、生成 prompt）
3. Cron 整合變得 trivial（直接在 cron payload 裡引用 SKILL.md 流程）
4. 不需要維護 JSONL 解析邏輯

**保留：**
- `validator.sh` — config 修改的 backup/rollback（確定性腳本比 agent 可靠）
- `risk-rules.json` — 風險分級規則（給 agent 參考的結構化資料）
- `fixes-history.json` — 修復歷史記錄

**移除：**
- `analyze.js`、`index.js`、`fix.js` — 邏輯轉移到 SKILL.md
- `lib/scanner.js`、`lib/patterns.js`、`lib/fixer.js` — agent 直接做

### 新目錄結構

```
self-evolution/
├── SKILL.md                # 主入口：完整分析和修復流程
├── README.md               # 專案說明
├── lib/
│   ├── risk-rules.json     # 風險分級規則（結構化）
│   └── validator.sh        # config backup/rollback 腳本
├── snapshots/              # 每日健康快照
│   └── YYYY-MM-DD.json     # 結構化分析結果
├── pending-actions.md      # high-risk 待人工確認
└── fixes-history.json      # 修復歷史
```

## 實作計劃

### Phase 0 — 架構重寫（P0，工作量：小）

**目標：** 把現有分析邏輯翻譯成 SKILL.md 指令

**具體工作：**
1. 撰寫新的 SKILL.md，定義完整的 analyze / fix / trend 流程
2. 建立 `risk-rules.json`，從 `classifier.js` 提取規則
3. 將 `validator.js` 改寫為 `validator.sh`（純 shell，不依賴 Node.js）
4. 定義 snapshot JSON schema
5. 封存舊的 Node.js 程式碼（移到 `legacy/` 或刪除）

**Snapshot JSON Schema：**
```json
{
  "date": "2026-03-12",
  "sessionsScanned": 15,
  "errors": {
    "tool_error": { "count": 3, "tools": ["read", "exec"], "deduped": true },
    "exec_failure": { "count": 1, "tools": ["exec"] },
    "timeout": { "count": 0 }
  },
  "totalErrors": 4,
  "topIssues": [
    { "type": "tool_error", "tool": "read", "pattern": "ENOENT", "count": 2, "risk": "low" }
  ],
  "recommendations": [
    { "priority": "high", "type": "missing_file", "description": "...", "risk": "low_risk" }
  ],
  "comparedToPrevious": {
    "errorDelta": -2,
    "newIssues": [],
    "resolvedIssues": ["timeout on web_fetch"]
  }
}
```

### Phase 4A — 資料管線重建（P0，工作量：中）

**目標：** 用 OpenClaw 原生工具取得 session 資料

**SKILL.md 中的分析流程：**
```
1. sessions_list(limit=20) → 取得最近 20 個 session
2. 對每個 session：sessions_history(sessionKey, limit=50, includeTools=true)
3. 掃描 toolResult 中 status === "error" 的條目
4. 按 error type + tool name 分組計數（去重：同 session 同 tool 同 error message 只計 1 次）
5. 與上一次 snapshot 比較（讀 snapshots/ 目錄最新檔案）
6. 輸出結構化 JSON 到 snapshots/YYYY-MM-DD.json
7. 生成人類可讀的摘要報告
```

**去重規則：**
- 同一個 session 內，相同的 `errorType + tool + errorMessage` 只計 1 次
- 跨 session 的相同錯誤各計 1 次（代表問題確實重複出現）

**時間窗口：**
- 預設分析最近 24 小時的 sessions
- 可指定 `--days N` 擴大範圍

---

### Phase 4D — 閉環修復（P1，工作量：中）

**目標：** low-risk 問題自動修復，high-risk 通知人類

**風險分級規則（risk-rules.json）：**

```json
{
  "low_risk": {
    "types": ["missing_file", "workspace_file", "cron_prompt"],
    "description": "只影響 workspace 檔案，不影響系統運行",
    "action": "auto_fix"
  },
  "high_risk": {
    "types": ["js_patch", "system_service", "config_change", "binary"],
    "paths": ["/opt/homebrew/lib/node_modules/openclaw/", "node_modules/"],
    "patterns": ["gateway restart", "systemctl", "brew services"],
    "description": "影響系統核心或需要 restart",
    "action": "notify_human"
  }
}
```

**修復流程：**
```
1. 讀取最新 snapshot 的 recommendations
2. 對照 risk-rules.json 分級
3. Low-risk:
   a. sessions_spawn 建立 isolated session
   b. 給定明確的修復任務 prompt
   c. 等待完成，檢查結果
   d. 記錄到 fixes-history.json
4. High-risk:
   a. 寫入 pending-actions.md
   b. 通知 Oliver（透過 message tool）
```

**安全機制：**
- 所有修復在 isolated session 執行
- 涉及 config 的修改先 backup（validator.sh）
- 修復失敗自動回滾
- 每次修復記錄完整 audit trail

---

### Phase 4B — Cron 自動化（P1，工作量：小）

**目標：** 整合進現有的每日進化 cron job

**方案：** 不新建 cron job，而是在現有的「每日進化檢查（09:00）」cron 中加入 self-evolution 分析步驟。

**修改後的 cron 流程：**
```
原有：代碼快照 → 發現需求 → 對外研究 → 整合報告
新增：代碼快照 → 健康分析（self-evolution）→ 發現需求 → 對外研究 → 整合報告
```

**健康分析步驟：**
1. 按 SKILL.md 的 analyze 流程跑一次
2. 如果有 low-risk 問題，自動修復
3. 如果有 high-risk 問題，寫入進化報告
4. 趨勢數據納入報告的「代碼狀態」區塊

---

### Phase 4C — 趨勢追蹤（P2，工作量：小）

**目標：** 追蹤系統健康趨勢

**實作：**
- 每次分析存 snapshot 到 `snapshots/YYYY-MM-DD.json`
- 分析時讀取最近 7 天的 snapshots
- 計算 delta：錯誤增減、新問題、已解決問題
- 報告格式範例：

```
📊 系統健康趨勢（最近 7 天）
錯誤率：📉 下降 23%（12 → 9）
已解決：timeout on web_fetch, ENOENT config.json
新問題：exec permission denied（low-risk，已自動修復）
```

**Snapshot 保留策略：**
- 最近 30 天：每日保留
- 30 天以上：自動清理

## 風險和注意事項

1. **Token 消耗** — SKILL.md 驅動意味著每次分析都用 agent token，需控制 session 數量和 history 長度
2. **分析品質** — agent 分析不如確定性程式碼穩定，需要在 SKILL.md 中給足夠清晰的指令
3. **Snapshot 膨脹** — 30 天 JSON 檔案不大，但要設定清理機制
4. **修復安全** — low-risk 自動修復仍需嚴格定義邊界，避免意外改動
